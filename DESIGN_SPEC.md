# UI Spec (по макетам) — зафиксировано

Этот файл фиксирует **то, что мы уже подробно обсудили и согласовали** по дизайну и поведению.

## Общие правила

1) **Дизайн не меняем** — цель сделать 1-в-1 по скринам.
   - Допустимое отклонение, которое обсуждалось: **кнопка “Обработать” может быть визуально неактивной** (disabled-стейт) когда нельзя запускать.

2) **Скругления (радиусы)**
   - Внутри левой и правой карточки **у всех объектов одинаковый радиус**.
   - У контейнеров (включая скролл-контейнер) — тот же радиус.

3) Фокус на UX: “не мельтешить”, всё предсказуемо.

## Экран загрузки (ImageUploader)

Файл/компонент: `frontend/src/components/ImageUploader.tsx`

### Поведение при импорте изображений (важно)

Требование: **при импорте не должно сразу вылезать изображение на главный (большой) экран**.

Причина: чтобы у пользователя была возможность сначала выбрать, и **ничего не мельтешило перед глазами**.

Согласованная логика:
- Импорт добавляет изображения в список миниатюр.
- Большое превью **не переключается автоматически** на только что импортированный файл.
- Пользователь **сам** выбирает нужную миниатюру кликом — только после этого меняется большое превью.

Текущее состояние реализации:
- Автовыбор первого импортированного изображения отключён.
  - (Раньше было: `setActiveId(prev => prev || next[0].id)`)
  - Сейчас: activeId **не меняется** при добавлении файлов.

### Скролл миниатюр — внешний вид и геометрия (самое важное)

Требование: скролл выглядит как в макете:
- Скролл происходит **внутри отдельного контейнера** слева.
- **Ширина контейнера скролла = ширине превью**.
- **Высота контейнера скролла = 3 превью** (вариант 1), то есть:
  - видно ровно 3 миниатюры,
  - и **видны** промежутки между ними,
  - поэтому высота = `3 * высота_превью + 2 * gap`.

Также критично:
- При скролле фото **не просто скрывается прямоугольником**, а **обрезается со скруглением по краям**.
  - Технически это означает: внешний контейнер имеет `overflow: hidden` + `border-radius`,
  - а внутренний — `overflow-y: auto`.

Текущее состояние реализации (Tailwind-классы):
- Контейнер клиппинга:
  - `rounded-2xl overflow-hidden`
- Внутренний скролл:
  - `overflow-y-auto overflow-x-hidden`
- Геометрия:
  - ширина = `w-20 md:w-24`
  - высота (ровно под 3 превью + 2 gap):
    - mobile: `h-[264px]` = `80*3 + 12*2`
    - md: `md:h-[320px]` = `96*3 + 16*2`

## Экран результата (Results / “готовое”)

Файл/компонент: `frontend/src/components/ResultsPage.tsx`

### Семантика сравнения (строго)

Это НЕ “оригинал/готовое” как подписи, а **“Было / Стало”** по смыслу:
- **Слева / Было**: фото **до обработки** (если сбита перспектива — то выпрямленная версия).
- **Справа / Стало**: уже готовое **векторное**.

Маппинг URL (как договорились):
- before (Было): `warped_original_url || original_url`
- after (Стало): `vector_preview_url || processed_url`

### Кнопки “Экспорт / Заново” — геометрия

Требование: две кнопки выглядят как сегментированный контрол.

Правило радиусов:
- **Внешние** радиусы (у “Экспорт” слева, у “Заново” справа) — **такие же, как у всех элементов внутри карточек**.
- **Внутренние** радиусы (стык двух кнопок) — **очень маленькие, но скруглённые**.

## Что ещё нужно добрать из макетов

Чтобы перенести **все цвета/эффекты 1-в-1** (тени, бордеры, фон, состояния hover/active/disabled, типографика), нужны все 5 скринов как файлы в репозитории.

Сейчас в корне проекта точно найдены:
- `скролл вверх.png`
- `скролл вниз.png`
- `готовое.png`

Если есть ещё 2 скрина, но они не находятся поиском, вероятные причины:
- нестандартное расширение/регистр,
- лежат в другой папке,
- или имеют другое имя.
